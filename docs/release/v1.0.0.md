# Briefed 1.0.0 — Release Notes

**Tag:** `v1.0.0`
**Date:** 2026-04-25

Briefed is a personal AI email agent that runs a daily pipeline on a
Gmail inbox: it ingests new mail, classifies each email into one of
four buckets (must-read, good-to-read, ignore, waste), summarizes the
must-read pile, extracts job postings into a filterable list, and
recommends senders to unsubscribe from. It never acts on the user's
behalf (ADR 0006 — recommend-only).

## Architecture (canonical decisions, plan §20.1)

- AWS Lambda + SnapStart for compute (api via Mangum, worker via SQS,
  fan-out via EventBridge Scheduler).
- Gemini 1.5 Flash primary; Claude Haiku 4.5 fallback gated to 100
  calls/day (ADR 0002).
- `LocalAuthProvider` only — argon2 password + TOTP MFA. The
  `AuthProvider` protocol seam stays open for a 1.2+ Cognito adapter.
- Two customer-managed KMS CMKs: `alias/briefed-token-wrap` for OAuth
  refresh tokens (§20.3); `alias/briefed-content-encrypt` for summary
  bodies, classification reasons, job match reasons, and unsubscribe
  rationales (§20.10). Per-row DEK; encryption context binds
  `{user_id, table, row_id}`.
- Polling for manual-run progress (TanStack Query
  `refetchInterval`) — no SSE in 1.0.0 (§20.5).

## Migration policy

Forward-only in prod. Additive columns first, data backfill next, drop
old columns in the *following* minor — never in the same release. The
Phase 9 [`0007_phase9_release_metadata`](../../backend/alembic/versions/0007_phase9_release_metadata.py)
migration follows this policy: pure additive, fully reversible in dev.

## Audit trail

Every prod deploy writes a `release_metadata` row capturing
`version`, `git_sha`, `alembic_head`, `api_schema_version`,
`db_schema_version`, `frontend_build_id`, `prompt_bundle_version`,
and `notes`. A rollback emits a *new* row pointing at the previous
SHA — rows are never updated.
