Briefed

A personal AI email agent. Every day it reads your Gmail, decides what matters, summarizes it, finds job openings, and tells you what to unsubscribe from — and it never touches your mailbox without you.

Python · FastAPIPydanticReact · Vite · PWA OpenRouter (Gemini Flash + Claude Haiku 4.5)Gmail API Supabase PostgresAWS Lambda (container image)SQSKMSTerraform
Version: 1.0.0 (released 2026-04-25) Audience: new engineers onboarding to the codebase Doc generated: 2026-05-30 Source of truth: code at commit dcf4284

01 Executive summary

Briefed is a single-user, self-hosted assistant that turns a noisy Gmail inbox into a short, useful daily briefing. You connect your Gmail account once; after that, Briefed quietly does the reading for you and shows the results in a phone-friendly web app.

📥 What it does

Sorts every new email into buckets (must-read, good-to-read, ignore, waste), writes plain-language summaries of the important ones, groups newsletters into a tech-news digest, pulls structured job listings out of recruiter emails, and ranks senders you should probably unsubscribe from.

⚙️ How it does it

A scheduled job wakes up once or twice a day, fetches new mail through the Gmail API, and pushes each email through a chain of small background workers. Each stage calls a large language model (LLM) through OpenRouter to read and reason, then stores the result in a database. The web app reads those results.

🛡️ What it will not do

It never archives, deletes, marks-read, replies, or unsubscribes for you. Release 1.0.0 is recommend-only — every action is a suggestion with an "Open in Gmail" link, so a wrong guess can never lose your mail.

The mental model in one paragraph

Think of Briefed as a tiny factory that runs overnight. A timer (AWS EventBridge Scheduler) starts the line. A fan-out step creates one work order per connected mailbox. That work order flows down a conveyor belt of queues (AWS SQS): first ingest the new emails, then classify them, then in parallel summarize the important ones and run a periodic unsubscribe hygiene pass. Each station is a small serverless function (AWS Lambda) that shuts off when idle, so the factory costs almost nothing when nothing is happening. The finished briefing lands in a Postgres database, and a React web app (installed like a phone app) shows it to you.

📌 Why this shape?

The whole design optimizes for one person, low cost, and safety. Serverless means you pay per-use (near zero at idle). Recommend-only means a model mistake can't damage your inbox. Two managed services (Supabase for the database, OpenRouter for the LLM) keep the operational surface tiny. The trade-offs behind each of these are written down as Architecture Decision Records.

02 How it works — a day in the life

Here is the same system told as a story, so the moving parts have a reason to exist before you meet them by name.

  1. The timer ticks — every 15 minutes. EventBridge Scheduler triggers the fan-out Lambda on a fixed 15-minute cadence. The cadence is constant; who actually runs is decided next. (You pick once-daily or twice-daily with specific HH:MM slots; those live on your profile.)
  2. Fan-out decides who is due. A cheap SQL filter shortlists candidate users, then a Python predicate (app.core.scheduling.is_due) authoritatively re-checks each one — handling timezone, DST, a "ran in the last hour" debounce, and stale locks. For each due user it takes a per-user idempotency lock (users.current_run_id) so two ticks can't stomp on each other, then drops one "go ingest this mailbox" message onto the ingest queue per active auto-scan account.
  3. Ingest fetches new mail. The ingest worker uses the saved sync cursor (Gmail's historyId) to ask Gmail only for what changed since last time. It de-duplicates, splits each message into lightweight metadata + a separately-encrypted body, and saves it. Then it chains the emails onto the classify queue.
  4. Classify decides what matters. For each email, a fast rule engine runs first (cheap, deterministic). If the rules are confident, it stops there. If not, it asks the LLM. The verdict — one of must_read, good_to_read, or ignore — is saved, with the model's reasoning encrypted.
  5. The line splits. Emails flagged important go to summarize; newsletters get grouped into tech-news clusters and summarized as a set.
  6. Hygiene runs. Periodically, the unsubscribe worker looks back 30 days per sender, scores how often they email you versus how often you engage, and ranks unsubscribe suggestions.
  7. You open the app. The React PWA calls the API and shows today's digest, the triage buckets, tech news, and unsubscribe recommendations. Because it's a PWA, the last digest is also readable offline.
✅ Idempotent by design

Every step is safe to re-run. Gmail fetches are keyed on historyId (re-runs are no-ops), each pipeline message carries a run_id, and rows like classifications and summaries are upserted per email. If a worker crashes, SQS redelivers the message and nothing is corrupted.

The big idea: one image, two "shapes", event-driven glue

There is one backend codebase and one container image. An environment variable, BRIEFED_RUNTIME, decides which entrypoint runs. The web traffic path (CloudFront → API Lambda) is completely separate from the work path (Scheduler → fan-out → SQS → worker Lambda), but both share the same code, the same database, and the same encryption keys. This is why a feature can be written once and used both by an API request and by a background worker.

🌐 The request path (interactive)

You hit CloudFront. Static PWA files come from S3. Anything under /api or /oauth is forwarded to the API Lambda, which runs FastAPI wrapped by Mangum. It reads/writes Postgres and decrypts content through KMS as needed.

🔁 The work path (batch)

The Scheduler fires the fan-out Lambda. From there it's all SQS messages and the worker Lambda, one queue per pipeline stage. Failed messages retry up to 5 times, then land in a dead-letter queue for inspection.

⚡ SnapStart: designed-for, but currently OFF

ADR 0003 planned for AWS SnapStart (snapshot a booted Lambda, restore in ~200–300 ms instead of a 1–3 s cold start). In the deployed Terraform it is intentionally disabled because all three Lambdas ship as container images (package_type = "Image"), and AWS SnapStart does not support container-image packaging. So cold starts are not yet mitigated. The code is still written to be snapshot-friendly — settings are hydrated from SSM and logging is configured at import time, not in a factory — so SnapStart can be flipped on if AWS adds container support or the team moves to ZIP packaging. The same reasoning is why heavy libraries (boto3, httpx, the LLM SDK) are imported lazily inside handler bodies; the per-file ruff ignore PLC0415 marks those intentional late imports.

Failure handling & graceful degradation

The pipeline is built to degrade rather than drop work. Two patterns repeat: retry-by-bubbling (let the exception propagate so SQS redelivers, eventually to the DLQ) and degrade-in-place (write a safe fallback row so the email is never silently lost).

StageWhat happens when it goes wrong
IngestIf the saved Gmail cursor is stale/rejected, it falls back to a bounded newer_than:14d scan instead of a full history walk. Token-refresh, quota, and provider errors are allowed to bubble so SQS retries the message.
ClassifyIf the LLM call fails entirely, the email is persisted as needs_review (graceful degradation) rather than dropped. Even on success, a model confidence below 0.55 is forced to needs_review so low-confidence guesses surface for a human instead of mislabeling.
SummarizeA missing prompt-version row is retried. LLM/parse failures are handled as service-layer outcomes; a missing source email is treated as a no-op so a deleted email can't wedge the queue.
UnsubscribeIf the borderline LLM call fails for one sender, that sender is skipped and the 30-day aggregate continues for the rest. User dismissals are always preserved across re-runs.
Any stageAfter 5 failed receives the message redrives to the shared DLQ (14-day retention). The worker reports partial batch failures, so one poison record doesn't force healthy records in the same batch to reprocess.
⚠️ Reliability knobs (from the SQS Terraform module)

Queue visibility timeout is 900 s (≥ the worker's max runtime). A message is retried up to 5 times before redriving to the shared dead-letter queue, which retains failures for 14 days. Normal queues retain messages for 4 days. A CloudWatch alarm watches DLQ depth.

🔀 Fallback chain

The chain is catalog-driven: [primary, *fallbacks] = Gemini 2.0 Flash then Claude Haiku 4.5, both routed through OpenRouter (ADR 0009). Adding a model is a one-line edit in catalog.py.

🔁 Retries & circuit breaker

Up to 3 attempts with exponential backoff + jitter, only on errors marked retryable. After 5 consecutive failures the breaker trips and the fallback takes over while it's open.

💰 Hard cost caps

A per-day USD cap (daily_llm_usd_cap) is enforced from each call's reported usage.cost; exceeding it raises LLMBudgetExceededError and opens a global breaker until UTC midnight. Claude Haiku is separately capped at 100 calls/day.

🧾 Full auditing

Every call writes one prompt_call_log row — model, tokens (incl. cache read/write), cost, latency, status (ok/fallback/error/skipped), and a redaction count summary (never the values).

Prompts are versioned artifacts, not strings

Prompts live in packages/prompts/ as versioned bundles with JSON Schemas, and are referenced by (name, version) — e.g. ("triage", 2), ("summarize_relevant", 1), ("category_digest", 1), ("newsletter_group", 1), ("unsubscribe_borderline", 1). At process start the PromptRegistry loads them into the prompt_versions table, which every audit row foreign-keys to. You never paste a prompt into business logic.

07 Security & encryption

Briefed handles your email, so privacy is a first-class design constraint, not an afterthought. Three layers matter: authentication, encryption at rest, and prompt redaction.

Authentication — a signed cookie

Because 1.0.0 is single-user self-host, auth is deliberately simple: a cookie named briefed_session containing the owner's user_id, signed with HMAC-SHA-256 using session_signing_key (see backend/app/api/session.py). Tampering breaks the signature and is rejected. The cookie is HttpOnly, SameSite=Lax, and Secure outside local dev, with a 30-day lifetime. The OAuth flow uses a second short-lived signed cookie (briefed_oauth_state, 10-min TTL) carrying the random state and the PKCE verifier. All /api/v1/* routes require the session cookie; only /health and the two OAuth endpoints are open.

⚠️ CSRF posture (be precise here)

CSRF defense in 1.0.0 rests on SameSite=Lax cookies + same-origin serving (CloudFront proxies the API under the PWA's origin). There is no separate CSRF token / double-submit token in the backend today. If a future change moves the API cross-origin or relaxes SameSite, add an explicit CSRF token for state-changing requests.

Encryption at rest — two KMS keys, envelope encryption

Supabase can technically read any column you don't encrypt, so the sensitive ones are encrypted before they ever reach the database (ADR 0008). Briefed uses two separate customer-managed KMS keys so a token leak and a content leak have independent revocation paths:

🔑 alias/briefed-*-token-wrap

Wraps your Gmail OAuth access & refresh tokens (oauth_tokens.access_token_ct / refresh_token_ct). Helper: app/core/security.py.

🔒 alias/briefed-*-content-encrypt

Wraps email-derived content: summary bodies, classification reasons, job rationales, unsubscribe rationale, email body excerpts. Helper: app/core/content_crypto.py.

The pattern is envelope encryption: a fresh 256-bit data key (DEK) encrypts the value with AES-GCM; KMS wraps the DEK; key material never leaves KMS. Every encrypt/decrypt binds the user_id into the KMS encryption context, so a ciphertext stolen for one user can't be decrypted under another. Revoking the Lambda role's kms:Decrypt permission instantly bricks all reads — a real trust boundary, not "security theater."

🔐 What "_ct" columns mean

Any column ending in _ct (ciphertext) holds an envelope-encrypted value — Supabase only ever sees opaque bytes. Repositories encrypt on write and decrypt on read; plaintext such as a summary body or an LLM rationale never touches disk. In the table reference below these are flagged encrypted.

Prompt redaction — shrinking the blast radius

Before any email text is sent to OpenRouter, it passes through a sanitizer chain (ADR 0010): identity → regex → Presidio. The identity scrubber replaces the user's own email/name/aliases with stable placeholders; regex catches residue; Presidio (Microsoft's PII NER) fills in names and contact info the regex misses. Honest scope: this reduces what a downstream breach or subpoena would expose; it does not hide that a request happened, its timing, or behavioral patterns. Re-identification of model output is off by default and gated behind an allowlist. The audit log stores only {kind: count}, never the reversal map.

13 Configuration & environment

All configuration flows through one typed object: Settings (pydantic-settings) in backend/app/core/config.py, accessed via the memoized get_settings(). Never call os.getenv in business logic — route through Settings so tests and SSM hydration keep working. Locally, values come from process env + a .env file. In Lambda runtimes, secrets are pulled from SSM Parameter Store at cold start and merged in (env still wins over SSM for local overrides).

🔑 Required secrets fail init hard

16 Assumptions, constraints & decisions

Briefed makes deliberate trade-offs. Knowing them up front prevents "why isn't this real-time?" and "why no auto-archive?" surprises. Each is backed by an Architecture Decision Record (ADR) in docs/adr/.

Core assumptions baked into 1.0.0

👤 Single user, self-hosted

One owner per deployment. Auth is a signed cookie, not a multi-tenant identity system. Many simplifications follow from this (no RLS, simple redaction policy, one set of KMS keys).

🛡️ Recommend-only (ADR 0006)

Briefed never mutates your mailbox. OAuth stays at gmail.readonly. Worst case of a model mistake is a bad suggestion, never lost mail.

⏱️ Daily, not real-time (ADR 0001)

Scheduled incremental sync via historyId; no Gmail push/webhook. Up to ~24h latency between an email arriving and being surfaced — fine for a daily digest.

💸 Cost-bounded (~$8–11/mo)

Serverless + free-tier database + capped LLM spend. A per-day USD cap and a Haiku call cap stop runaway bills. Steady-state AWS is ~$1–2/mo; the two KMS keys add ~$2/mo.

🤖 Quality over auto-escalation (ADR 0002→0009)

Gemini Flash does the work; Claude Haiku is a gated fallback. No automatic jump to a pricier model — quality regressions page a human via the eval suite instead.

📱 PWA only (ADR 0005)

No native iOS/Android app. iOS Safari quirks are accepted: progress is polled (no long-lived sockets), OAuth escapes the in-app webview, IndexedDB eviction is handled with Dexie.

Trust boundaries (read this before touching prompts or crypto)

🌐 The OpenRouter hop (ADR 0009)

Email content leaves your infrastructure and passes through OpenRouter on the way to Gemini/Anthropic. Briefed sends data_collection: "deny" and runs the redaction chain first, but the request body is plaintext to OpenRouter's edge. This is knowingly accepted for a single-user personal project and is explicitly not acceptable for PHI/PII at scale or multi-tenant use. Onboarding a second user requires revisiting ADR 0009.

🔐 The database trust boundary (ADR 0004 + 0008)

Supabase can read any column you do not encrypt. Therefore tokens and all email-derived content are envelope-encrypted with KMS before storage; Supabase only ever sees metadata (labels, scores, timings) and ciphertext. Never add a plaintext content column without an ADR.

🧹 What redaction does not do (ADR 0010)

Redaction reduces blast radius; it is not invisibility. It does not hide that a request happened, its timing/size, or behavioral patterns inferable from prompts. Re-identifying model output is off by default and allowlist-gated. Treat redaction as harm-reduction, not a guarantee.

Explicit non-goals for 1.0.0

  • No mailbox writes (archive, delete, mark-read, send, auto-unsubscribe).
  • No real-time / push ingestion.
  • No multi-tenant or hosted SaaS mode — single user only.
  • No providers other than Gmail (though the MailboxProvider interface (ADR 0007) makes Outlook/IMAP additive, not a rewrite).
  • No auto-generated ERD pipeline — the data model is hand-maintained alongside migrations.

Known limitations & residual risks (as of this commit)

Honest gaps a new engineer should know before relying on a behavior. None are bugs — they're conscious 1.0.0 trade-offs or "not built yet" edges.

LimitationDetail & impact
Metadata is plaintextSubjects, snippets, sender/recipients, labels, and timestamps are stored unencrypted (only content + rationales are _ct encrypted). Accepted residual risk; a database read still reveals who emailed you and when.
LLM caps are per-processThe daily USD cap and the Haiku call cap are tracked in-memory per warm Lambda. Across many concurrent instances the global cap is approximate, not strict. For hard global enforcement, move budget state to the database or a strongly-consistent store.
Scheduled runs aren't ledgeredOnly the manual POST /api/v1/runs path writes a digest_runs row. The scheduled fan-out propagates a run_id lock but does not insert a digest_runs row, so GET /history reflects manual runs as of this commit. Add explicit scheduled-run ledgering if history must cover scheduled runs.
SnapStart is offContainer-image packaging means cold starts are not mitigated (see §4). The code is snapshot-ready for when this changes.
OAuth id_token not signature-verifiedThe callback decodes the Google id_token for the email/sub claims but defers cryptographic signature verification to a later phase — it trusts the response because it came directly from Google's HTTPS token endpoint.
This doc needs network for diagramsMermaid is loaded from a CDN. Offline, the diagram source remains readable as text but won't render as SVG.

17 Glossary

BriefedThis project — a personal AI email agent that produces a daily inbox briefing.
PWAProgressive Web App — a website installable to the home screen with offline support.
Fan-outThe step that turns "run the daily job" into one work item per connected mailbox.
SQSAmazon Simple Queue Service — the message queues between pipeline stages.
DLQDead-Letter Queue — where messages go after failing the configured number of retries (5).
SnapStartAWS Lambda feature that snapshots a booted function to cut cold-start latency. Designed-for here but currently disabled because the Lambdas ship as container images (see §4).
MangumAdapter that lets a FastAPI (ASGI) app run inside AWS Lambda.
PKCEProof Key for Code Exchange — an OAuth hardening step; Briefed generates a verifier/challenge pair so an intercepted auth code can't be redeemed without the verifier.
Partial batch responseAn SQS+Lambda feature where the worker reports only the failed records in a batch, so healthy records aren't reprocessed.
is_due predicateThe Python check (app.core.scheduling) that decides whether a user's schedule is due on a given 15-minute tick — timezone, DST, debounce, and stale-lock aware.
OpenRouterA single API that routes prompts to multiple LLM providers (here: Gemini Flash, Claude Haiku 4.5).
RubricThe set of user rules that classify emails before (or instead of) an LLM call.
Classification labelOne of must_read, good_to_read, ignore.
Envelope encryptionEncrypt data with a one-time data key (DEK), then encrypt the DEK with a KMS master key.
CMKCustomer-Managed Key in AWS KMS — Briefed uses two (token-wrap and content-encrypt).
Encryption contextExtra key/value data (here, user_id) bound into every KMS call; a mismatch is rejected.
_ct columnA ciphertext column — envelope-encrypted; the database never sees plaintext.
Sync cursorThe saved Gmail historyId that makes each fetch incremental and idempotent.
Circuit breakerA guard in LLMClient that stops calling a failing provider after 5 consecutive errors.
ADRArchitecture Decision Record — a short doc capturing one decision and its trade-offs.
Redaction chainidentity → regex → Presidio scrubbing applied to every outbound prompt.
Run / run_idOne execution of the daily pipeline; threaded through every message. Manual runs are ledgered in digest_runs (scheduled runs currently are not — see §16).
