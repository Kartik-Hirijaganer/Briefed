## Context

Briefed is a personal, self-hosted, open-source email triage and summarization agent that connects one or more Gmail accounts and — without manual triggering — ingests, classifies, summarizes, and delivers a daily digest. It separates *must-read / good-to-read / ignore / waste* buckets, splits **tech news** into its own digest, extracts **jobs** against user-configured filters, and recommends **unsubscribes**. It must run well on iPhone as a PWA with **offline viewing** of already-synced summaries.

This plan treats the product as a long-lived open-source codebase — not a hackathon — and optimizes for **maintainability, reliability, security, accuracy, and cost-consciousness**, in that order. The deployment target is AWS. The default user model is **single-user self-hosted** (each contributor deploys their own instance from the repo); the schema and access control are structured so a multi-user path is additive, not rewrite.

## 1. Executive Summary

**Product goal.** Compress a noisy multi-account inbox into a small, trustworthy daily artifact: a must-read list, a tech-news digest, a jobs board filtered to the user's criteria, and an unsubscribe queue. Users never delegate *action* (replies, deletions) to the agent in 1.0.0; they delegate *attention*.

**User value.** Hours per week saved skimming newsletters and recruiter email; a single daily surface instead of three inboxes; a job board that doesn't invent postings.

**Solution approach.** A single Python FastAPI codebase deployed as two Lambda functions (`api` via Mangum, `worker` via SQS event source) with SnapStart enabled, orchestrated by EventBridge Scheduler → fan-out Lambda → per-stage SQS queues (see §19.15). Classification + summarization + job extraction go through a provider-agnostic `LLMProvider` layer — canonical profile is **Gemini 1.5 Flash (paid, no-training tier)** primary + **Claude Haiku 4.5** gated fallback. A React 18 + Vite PWA with shadcn/ui + Tailwind v4 provides the dashboard and offline reader.

**Release 1.0.0 scope.** Production-grade single-user self-hosted deployment covering: multi-Gmail OAuth, scheduled ingestion, four-way triage + newsletter + job + unsubscribe pipelines, daily digest via SES + PWA web push, web/mobile UI with offline cache, observability + audit logs, CI/CD with atomic Lambda-alias rollback, and a documented install story for OSS contributors. **Target steady-state cost: ~$6–9/month total** (§19.15).

**Explicitly deferred from 1.0.0:** multi-tenant SaaS, non-Gmail providers (Outlook/IMAP), auto-reply or auto-delete actions, local on-device LLM, mobile app store submission, team/shared inboxes.

## 2. Product Goals and Non-Goals

### In scope (1.0.0)
- Connect ≥1 Gmail account via OAuth (offline access, refresh token).
- Scheduled daily ingestion (configurable cron per user).
- Four-way triage: **must_read / good_to_read / ignore / waste** + flags `newsletter`, `job_candidate`.
- Summaries for must-read + good-to-read + newsletters (grouped) + job postings.
- Configurable job filters (title keywords, location allow-list, remote flag, salary floor, seniority).
- Unsubscribe recommendations ranked by frequency × engagement × waste rate.
- Cross-account consolidated digest.
- React PWA: desktop + iPhone, responsive, offline viewing of cached digest/summaries, add-to-home-screen.
- Configuration UI for rubric, prompts, schedule, job filters, thresholds.
- Observability: structured logs, traces, per-run audit, cost metering.
- Open-source hygiene: LICENSE (MIT, present), README, CONTRIBUTING, sample `.env`, one-command local dev, CHANGELOG.

### Explicitly out of scope (deferred)
- Auto-replies, auto-deletions, auto-unsubscribes (we *recommend*, user acts).
- Outlook/IMAP/Fastmail connectors.
- Multi-user SaaS hosting, billing, quotas, admin UI.
- Mobile-native app (iOS/Android store submissions).
- Local/offline LLM (Ollama, llama.cpp).
- Push via APNS — iOS PWA push support is unreliable pre-iOS 16.4 standalone; we rely on email digest + in-app pull-to-refresh as the primary delivery channel.
- HIPAA controls (see §11 — explicitly out of scope and unnecessary for personal use).

## 3. Core Functional Requirements

| Requirement | 1.0.0 Behavior |
|---|---|
| **Multi-Gmail support** | `connected_accounts` table, N accounts per user, each with its own OAuth refresh token, quota budget, and sync cursor. |
| **Scheduled processing** | EventBridge Scheduler cron per user (default `0 7 * * *` in user's TZ); fan-out Lambda enqueues one ingest task per account. |
| **Triage/classification** | Rule-first (rubric_rules table) → `LLMProvider` fallback (Gemini Flash per §19.15) with Pydantic tool-use schema `TriageDecision`. |
| **Summarization** | `LLMProvider` per-email summaries; tech-news group-summary per topic cluster. |
| **Tech news digest** | Newsletters detected via `List-ID` + curated domains; clustered by topic (embeddings or simple keyword buckets); one Sonnet call per cluster. |
| **Job extraction** | `LLMProvider` structured extraction of `JobMatch` fields; DB-driven `job_filters` decide `passed_filter`. |
| **Unsubscribe recommendations** | SQL aggregation over rolling 30 d (frequency × waste rate × open rate); LLM only on borderline cases. |
| **Inbox hygiene guidance** | Quick stats card on dashboard: count + % waste/ignore per sender + link to unsubscribe queue. |
| **Mobile/iPhone** | PWA manifest, viewport-fit=cover, bottom tab bar <md, safe-area insets, swipe gestures on email cards. |
| **Offline** | TanStack Query cache persisted to IndexedDB via async-storage-persister; mutations queued and replayed on reconnect. |
| **Configurable preferences** | All thresholds, cron, rubric, job filters, topics, delivery channels stored in `user_preferences` + `rubric_rules` + `job_filters`; **nothing hardcoded** — config-driven throughout. |

## 4. Non-Functional Requirements

- **Security:** OAuth refresh tokens envelope-encrypted (AES-GCM + KMS-wrapped DEK); HttpOnly Secure SameSite=Lax session cookies; least-privilege IAM task roles; no secrets in code; Supabase RLS on user-owned rows.
- **Reliability:** at-least-once pipeline with idempotent handlers; per-stage SQS + DLQ; circuit breaker on LLM; partial-run status model (no silent failures).
- **Maintainability:** strict typing (mypy strict, TS strict + `exactOptionalPropertyTypes`), Pydantic v2 for all boundaries, Google-style docstrings/JSDoc enforced by lint, centralized UI design system in `frontend/src/ui/`.
- **LLM accuracy:** pinned prompt versions, per-version call logs, regression eval (Promptfoo) gating PRs that touch prompts, confidence scores with `needs_review` flag below threshold.
- **Performance:** classification inline at <300 ms p50; summarization batched overnight; dashboard first-render <1.5 s on 4G mobile; offline cold open <500 ms.
- **Cost efficiency:** target steady-state <$30/month (see §5 cost model). Batch API + prompt caching + rule short-circuits.
- **Auditability:** append-only `audit_logs`, every LLM call logged to `prompt_call_log` with token/cost attribution, run-level `digest_runs.stats`.
- **Extensibility:** provider-agnostic ingestion interface (`MailboxProvider` protocol) so Outlook/IMAP slot in without touching the pipeline.
- **Release readiness:** semver + Keep-a-Changelog + signed tags + blue/green deploy + one-command rollback.

## 6. LLM / AI Strategy

### Provider choice — see §19.15 (canonical)

**Canonical profile**: `GeminiProvider` (Gemini 1.5 Flash paid tier, ~$3.60/mo) primary + `AnthropicDirectProvider` (Claude Haiku 4.5, ~$1–2/mo gated) fallback. Full cost math and model tiering in §19.15.

`BedrockProvider` and `OpenRouterProvider` remain available via the `LLMProvider` protocol (§19.4) — swappable by `settings.llm.provider` config, no code change.

### Prompt caching strategy

Provider-agnostic: the `LLMProvider` adapter translates the shared `PromptSpec` into whichever caching mechanism the active provider exposes (Anthropic `cache_control`, Gemini context caching, Bedrock CachePoint). Caching savings are upside, not baseline — §19.15 cost math does not rely on them.

- System prompt + rubric + few-shots → cached prefix where supported.
- Email body → dynamic suffix (never cached).

Target cache-hit rate: **>40% on Gemini primary** (Gemini caching is less mature than Anthropic's 1-hour tier). Alert if <20% for 24 h.

### Pipeline

```
Gmail pull → dedupe/normalize → signal extraction → rules short-circuit
  → classify via LLMProvider (if needed)     [Gemini Flash primary in §19.15]
    → {waste, ignore} → store and stop
    → {newsletter}   → enqueue newsletter_group
    → {job_candidate}→ JobMatch extract (sync if user opens inline)
    → {must_read, good_to_read} → enqueue summary (sync if user opens inline)
  → unsubscribe scorer (SQL, LLM only on borderline)
```

### Structured outputs (Pydantic v2, tool-use `input_schema`)

All under `backend/app/llm/schemas.py`; all `ConfigDict(frozen=True, extra="forbid")`:

- `TriageDecision(category, confidence, rationale_short, is_newsletter, is_job_candidate)`
- `EmailSummary(tldr, key_points[], action_items[], entities[], confidence)`
- `JobMatch(title, company, location, remote, salary_range, apply_url, fit_score, match_reason, rejected_reason)`
- `UnsubscribeSuggestion(sender, confidence, rationale, list_unsubscribe_url)`

`extra="forbid"` fails loudly if the model invents fields.

### Prompt management

Versioned Markdown files with YAML frontmatter under `backend/prompts/{key}/v{n}.md`:

```yaml
---
key: triage
version: 3
model: claude-haiku-4-5
temperature: 0.0
max_tokens: 400
schema_ref: TriageDecision
cache_tier: extended_1h
---
```

A `PromptRegistry` loads all on startup, exposes `registry.get("triage", version=3) -> PromptSpec`. Every call logs to `prompt_call_log(prompt_key, version, model, tokens_in/out/cache_read/cache_write, latency_ms, cost_usd)`.

### Evaluation framework

- **Tool: Promptfoo** (+ thin pytest wrapper).
- **Golden set:** 80 hand-labeled emails (20 must / 20 good / 20 ignore / 10 waste + 15 newsletters + 10 jobs + 5 phishing) stored at `backend/eval/golden/*.json`.
- **Metrics + thresholds** (`backend/eval/thresholds.yaml`): triage macro-F1 ≥ 0.90; summary faithfulness ≥ 4.3/5 (Opus-as-judge rubric); job field P/R ≥ 0.85; ECE calibration < 0.08.
- **CI integration:** `eval-regression` GHA job triggers on PRs touching `backend/prompts/**`, `backend/app/llm/**`, or `backend/eval/**`; blocks merge on regression.

### Hallucination reduction + confidence

Three layers:
1. Model-reported confidence as required schema field; prompt permits `{category=ignore, confidence<0.5}` as an explicit "I don't know."
2. Heuristic corroboration — e.g., if `salary_range` doesn't regex-match body text, null it and lower `fit_score`.
3. Policy gates — triage < 0.55 → `needs_review`; job < 0.7 → suppress from digest; unsubscribe < 0.8 → never auto-act.

### PII redaction (optional, off by default)

Custom lean regex suite (SSN, Luhn-validated CCN, MRN, DOB) swappable to Presidio. Personal-use default is OFF — the user's own inbox going to Anthropic is no new exposure beyond Gmail. Contributors opt-in via `settings.llm.redact_pii=true`.

### Cost controls

- Per-account daily token budgets (soft + hard caps) on `connected_accounts.daily_budget_*`; at 80% soft cap, summarization pauses; at hard cap, all LLM pauses; banner surfaces in UI.
- Batch API for all non-urgent summarization.
- Rule short-circuit skips 20–30% of LLM calls.
- Cache-hit monitoring → alarm.

### Fallback + degradation

`backend/app/llm/client.py` exposes `LLMClient.call(spec, inputs)`:
1. Anthropic direct (3 retries, exp backoff 1/4/12 s with jitter).
2. OpenRouter equivalent (2 retries).
3. Enqueue to `deferred_llm_calls` and mark `digest_runs.status='partial'` with `degraded_reason`.

User-visible state is always explicit — never a silent half-digest.

## 7. Target Architecture

### One deployable, two roles

A single container image with two handler entrypoints — `lambda_api.mangum_handler` (FastAPI via Mangum) and `lambda_worker.sqs_dispatcher` (SQS event source). Both deployed as separate Lambda functions with SnapStart + distinct IAM roles. Shared code, shared schemas. See §19.15 for runtime specifics.

### Components and interactions

```
                  ┌──────────────────────────────┐
                  │ EventBridge Scheduler (cron)  │
                  └──────────────┬────────────────┘
                                 │ per-user cron
                                 ▼
                  ┌──────────────────────────────┐
                  │ fan-out Lambda (Python)       │
                  │  - reads enabled users/accts  │
                  │  - enqueues ingest msgs       │
                  └──────────────┬────────────────┘
                                 ▼
                       SQS: briefed-ingest
                                 │
                                 ▼
                  ┌──────────────────────────────┐
                  │ Worker Lambda (SnapStart)     │
                  │ handlers: ingest / classify / │
                  │ summarize / jobs / unsub /    │
                  │ digest / maintenance          │
                  └──┬──────────────────────────┬─┘
                     │ per-stage enqueue        │ writes
                     ▼                          ▼
         SQS: classify/summarize/…      Supabase Postgres
                                                │
                                                │
              S3 (raw MIME / digest HTML / pg backups)
                                                │
        ┌───────────────────────────────────────┘
        │ SES email + pywebpush
        ▼
      User's inbox + PWA
                     ▲
  CloudFront + S3 ───┴── PWA (React) ──── API Lambda (Mangum + SnapStart)
```

**Component responsibilities:**

- **Gmail integration layer** (`backend/app/services/gmail/*`) — owns OAuth + refresh, `historyId`-based incremental sync, quota-aware token-bucket fetcher, batched `messages.get`, MIME→`EmailMessage` Pydantic parser. Single validated boundary; downstream code never sees raw Gmail JSON.
- **Scheduler / orchestration** — EventBridge Scheduler → small fan-out Lambda → `briefed-ingest` SQS. Lambda reads `users` + `connected_accounts` + `user_preferences.digest_cron` and enqueues one message per `(user_id, account_id, run_kind)`. Materializes a `digest_runs` row.
- **Ingestion pipeline** (`backend/app/services/ingestion/`) — pulls new message IDs since `sync_cursors.history_id`, fetches MIME in batches, optionally writes raw MIME to S3 (gated by `user_preferences.store_raw_mime`), writes metadata to `emails`, emits `classify` messages. Dedup via `UNIQUE(account_id, gmail_message_id)` + content hash.
- **Classification pipeline** (`backend/app/services/classification/`) — rule engine (`rubric_rules`) runs first; on abstain, LLM call via `LLMClient` (provider per §19.15). Result → `classifications` + fan-out to summarize / jobs / unsubscribe paths.
- **Summarization pipeline** (`backend/app/services/summarization/`) — `relevant.py` for per-email summaries; `tech_news.py` clusters newsletters (simple keyword + sender-domain clustering for 1.0.0; revisit with embeddings in 1.1) and group-summarizes each cluster.
- **Job extraction pipeline** (`backend/app/services/jobs/`) — LLM extracts `JobMatch` from `job_candidate` emails via `LLMClient`; rule engine applies `job_filters.predicate` (JSONB) to decide `passed_filter`.
- **Unsubscribe recommender** (`backend/app/services/unsubscribe/`) — SQL aggregate (sender_domain × 30 d): frequency + engagement score + waste rate + `List-Unsubscribe` header presence. LLM called only on borderline (2-of-3 criteria).
- **Preferences service** (`backend/app/services/preferences/`) — single source of truth. All pipelines read via `PreferencesService.get(user_id)`. Versioned schema with `config_version` upcast path.
- **Digest delivery** (`backend/app/services/digest/`) — composer assembles Markdown + rendered HTML, writes to `s3://briefed-digests/<user>/<date>.html`, delivers via SES + pywebpush (best-effort).
- **Audit + observability** (`backend/app/services/audit/`, `backend/app/observability/`) — `audit_logs` append-only row per stage; structlog JSON; OTel spans; CloudWatch EMF metrics; Sentry for uncaught exceptions.
- **Frontend / mobile / offline layer** — React 18 PWA on CloudFront + S3; TanStack Query cache persisted to IndexedDB for offline read; session cookie auth; openapi-fetch for typed API calls. Talks only to its own FastAPI backend; never to Gmail/Claude directly.
- **Admin/settings layer** — all via `/api/v1/*` + `/settings/*` UI routes. No separate admin service.
- **Storage / caching** — Postgres for metadata; S3 for blobs; no Redis in 1.0.0.
- **Audit / observability (dashboards + alarms)** — CloudWatch dashboards for SQS depth, worker success/fail, LLM spend, Gmail quota, digest success; SNS → email on breach.

## 8. Data Model / Storage Design

Portable SQL — targets Supabase Postgres and RDS equally. Every table: `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`, `created_at`, `updated_at TIMESTAMPTZ NOT NULL DEFAULT now()` (trigger).

| Table | Purpose | Key columns |
|---|---|---|
| `users` | Account owners | `email CITEXT UNIQUE`, `display_name`, `tz`, `status CHECK in ('active','disabled','deleted')`, `last_login_at` |
| `connected_accounts` | Per-Gmail-account | `user_id FK`, `provider CHECK='gmail'`, `email CITEXT`, `gmail_account_id`, `status`, `daily_budget_in INT`, `daily_budget_out INT`, `UNIQUE(user_id, email)` |
| `oauth_tokens` | Envelope-encrypted | `account_id FK UNIQUE`, `access_token_ct BYTEA`, `refresh_token_ct BYTEA`, `dek_wrapped BYTEA`, `scope TEXT[]`, `expires_at` |
| `sync_cursors` | Incremental-sync state | `account_id PK FK`, `history_id BIGINT`, `last_full_sync_at`, `last_incremental_at`, `stale BOOLEAN` |
| `emails` | Email metadata (body in S3 optional) | `account_id FK`, `gmail_message_id`, `thread_id`, `internal_date`, `from_addr CITEXT`, `to_addrs CITEXT[]`, `subject`, `snippet`, `labels TEXT[]`, `list_unsubscribe`, `content_hash BYTEA`, `raw_s3_key`, `UNIQUE(account_id, gmail_message_id)`, GIN idx `labels`, idx `(account_id, internal_date DESC)`, **RANGE partitioned by `internal_date` monthly, 12 partitions retained** |
| `classifications` | Triage verdict | `email_id FK UNIQUE`, `label CHECK in (…)`, `score NUMERIC`, `reasons JSONB`, `rubric_version INT`, `prompt_version_id FK`, `model`, `tokens_in/out INT` |

**RLS (Supabase path):** `USING (user_id = auth.uid())` on every user-owned table. Worker uses service-role key that bypasses RLS — documented trust boundary. On RDS migration, move to session-var policies (`SET LOCAL app.user_id`) so the SQL stays portable.

## 10. UI / Design System Strategy

### Centralized, lint-enforced design system

All primitives live in `frontend/src/ui/`. Feature folders compose from `ui/` and **may not** redefine primitives. An ESLint `no-restricted-imports` rule fails any feature-level file that imports styled primitives from outside `ui/`. A Tailwind `no-arbitrary-values` rule blocks `p-[17px]` or `text-[#ff00aa]` at lint time — tokens only.

### Tokens (`frontend/src/ui/tokens.css`)

- **Colors:** `--color-bg / bg-muted / surface / border / fg / fg-muted / accent / accent-fg / danger / warn / ok`. Dark mode via `:root[data-theme='dark']` variable swap — zero classname churn.
- **Spacing:** `--space-1` (4 px) through `--space-16` (64 px) restricted to `4/8/12/16/24/32/48/64`.
- **Radii:** `sm / md / lg / full`. **Shadows:** `1 / 2 / 3`. **Motion:** `--motion-fast 120ms`, `--motion-base 200ms`, `--ease-standard cubic-bezier(.2,.0,.0,1)`.
- **Typography:** one family (`Inter Variable, system-ui`); ramp `--fs-xs…3xl` with paired `--lh-*`. Inputs min 16 px (iOS auto-zoom).

Tailwind v4's `@theme` block **mirrors these tokens** so Tailwind classes compile to the tokens. Tokens are the single source; Tailwind is the ergonomic alias.

### Primitives + discriminated-union prop shapes

- `<Button>` — union over variants `primary | secondary | ghost | destructive | link`. `link` requires `href`; types prevent a link button without one.
- `<Field>` (react-hook-form bound) with `control`, `name: Path<TValues>`, `label`, `description`, `render`. Owns a11y (`<label htmlFor>`, `aria-describedby`, `aria-invalid`) once.
- `<DataTable<TRow>>` — TanStack Table v8 + Virtual v3; one component for jobs + history; sticky headers, selection, slots for empty / loading / error.
- `<Dialog>`, `<Popover>`, `<DropdownMenu>`, `<Tabs>`, `<Toast>` — all Radix-backed; same motion tokens.
- Centralized: `<EmptyState>`, `<Skeleton>` (shape variants), `<ErrorState>`, `<Alert>` (`tone`), `<Badge>`, `<Card>`.

### Information architecture (routes)

- `/` — dashboard (cross-account consolidated summary: counts, must-read preview, unread-since, cost today).
- `/must-read`, `/good-to-read`, `/ignore`, `/waste` — share the triage list/card components.
- `/jobs` — job board with filter drawer + bulk actions.
- `/news` — tech-news digest grouped by cluster.
- `/unsubscribe` — ranked recommendations with bulk confirm.
- `/settings` (→ `/accounts | /preferences | /prompts | /schedule`).
- `/history` — runs table; `/history/:runId` detail with stage timeline and cost breakdown.
- `/login`, `/oauth/callback`.

### Mobile / iPhone UX

- `<AppShell>` with `useBreakpoint()`: sidebar ≥ md, `<BottomTabBar>` < md (5 tabs + overflow). `env(safe-area-inset-*)` padding.
- Viewport: `width=device-width, initial-scale=1, viewport-fit=cover` + `apple-mobile-web-app-capable`.
- Touch targets ≥ 44 px (Tailwind `minHeight.touch: 44px`).
- Pull-to-refresh on digest lists (custom hook, `overscroll-behavior: contain`).
- Swipe gestures: `framer-motion` `drag='x'` on `<EmailCard>` — left → mark-ignore, right → mark-must-read, with spring-back and color-reveal backgrounds.

### PWA + offline strategy

- **Workbox via vite-plugin-pwa**, `registerType: 'autoUpdate'`. App shell precached; runtime caching:
  - `GET /api/digest/**`, `/api/jobs`, `/api/news`, `/api/unsubscribe`, `/api/history` → `StaleWhileRevalidate`, 7 d / 100 entries.
  - `GET /api/summaries/:id` → `CacheFirst`, 30 d (immutable once generated).
  - Mutations → no HTTP cache; go through TanStack Query + sync queue.
- **TanStack Query persistence** — `persistQueryClient` backed by `@tanstack/query-async-storage-persister` + **Dexie** (see §19.10 — range queries on `pending_mutations` + indexed summary cache). `gcTime` = 7 d; per-query `staleTime`. Hydrates before first render for instant-offline-open.
- **Sync queue** — IDB store `pending_mutations {id, type, payload, createdAt, attempts}`. On `offline`: `onMutate` updates cache optimistically and enqueues. On `online`: drain FIFO with reconciliation (server-wins for canonical fields, client-wins for ephemeral UI fields). A `<QueuedActionsSheet>` lets the user inspect / cancel.
- **iOS PWA honest reality:**
  - Origin-storage quota in home-screen PWAs is nuanced — design for ~50 MB; cap persisted cache at ~15 MB via `navigator.storage.estimate()` with an 80% warning.
  - iOS may **evict IDB after 7 days of non-use** — handle "cached data cleared" state gracefully; resync on first online fetch.
  - **No Background Sync, no Periodic Sync, no Web Push without APNS provisioning.** 1.0.0 does NOT promise background delivery. Email digest is primary; in-app pull-to-refresh is secondary.
  - Add-to-Home-Screen is manual on iOS — ship a dismissible `<InstallPromptIOS>` primitive. On Chromium we capture `beforeinstallprompt`.
  - Use `100dvh` / `100svh` with fallbacks; avoid `100vh`.

### Auth flow

Backend owns Gmail OAuth end-to-end. Frontend posts to `/api/auth/google/start`; Google redirects to backend callback; backend sets `HttpOnly Secure SameSite=Lax` session cookie and 302s to `/oauth/callback?status=ok`. API client sends `credentials: 'include'`. 401 → `queryClient.clear()` → `/login`. **No tokens in JS, no localStorage auth anything.** CSRF via double-submit: readable cookie + `X-CSRF-Token` header on state-changing verbs.

### Shared types (contract codegen)

- `/make-docs` exports `docs/openapi.json` (already a command).
- Frontend script `npm run codegen` runs `openapi-typescript docs/openapi.json -o src/api/schema.d.ts`.
- CI step: `make-docs` + `codegen` + `git diff --exit-code` — drift fails CI.
- `frontend/src/api/client.ts` exports a typed `api` from `openapi-fetch`. Lint rule: `no-restricted-globals: ['fetch']` outside `src/api/`.

### Accessibility + i18n

- Radix primitives → correct roles + focus management for free.
- `:focus-visible` rings via `--focus-ring` token; `prefers-reduced-motion` respected in framer-motion; icon-only buttons require `aria-label` (custom lint).
- Color contrast verified WCAG AA in Storybook.
- `react-i18next`, English-only in 1.0.0; strings co-located per feature (`features/**/strings.en.json`); all JSX literals go through `t()`.

## 11. Security, Privacy, and Compliance

### HIPAA assessment — **explicitly not applicable, and claiming it would be misleading.**

HIPAA regulates Covered Entities (health plans, providers, clearinghouses) and their Business Associates handling Protected Health Information in professional contexts. A personal self-hosted email client reading the user's own inbox — even a healthcare professional's — is **not** a HIPAA-regulated workflow. There is no BAA with the user (you are the user), no disclosure to third parties in the HIPAA sense, and the Gmail-of-record remains the system of record.

Where HIPAA **would** become relevant and this project does NOT offer those guarantees:
- Offering Briefed as a multi-tenant SaaS to Covered Entities — requires BAAs with Anthropic, AWS, Supabase, Google; formal risk analysis; full administrative safeguards.
- Ingesting a work mailbox governed by an employer HIPAA policy (even then, the organization's BAA is the operative instrument, not this tool's architecture).

**We will be explicit in the README:** "Briefed is security-first but is **not** HIPAA-compliant and is not intended for processing PHI in a regulated context. If you handle PHI, do not run Briefed against that mailbox."

### Security-first design (required regardless of HIPAA)

- **Gmail OAuth scopes** — request the minimum: `gmail.readonly` + `gmail.metadata` (and `openid email profile` for identity). Never ask for `gmail.modify` unless/until a future release explicitly acts on mail.
- **Token storage** — envelope encryption: random 256-bit DEK per token, AES-GCM encrypted ciphertext in Postgres; DEK wrapped by a master key stored in SSM Parameter Store SecureString (§19.15); DEK rotation on re-auth; master key rotated quarterly by the maintenance job (§19.13). `oauth_tokens` table never leaves the DB in plaintext; backups use the same wrapped keys.
- **Secret management** — AWS SSM Parameter Store SecureString for Gemini + Anthropic + Google client secret + DB URL + VAPID + session-signing key + token-wrap key (§19.15). 10k parameters free forever. Quarterly rotation via maintenance job. **No secrets in `.env.example` — only placeholders.**
- **Least privilege** — separate IAM roles for `api` Lambda vs `worker` Lambda; each role scoped to the exact SQS queues, S3 prefixes, and SSM parameter paths needed.
- **Audit logging** — append-only `audit_logs` partitioned monthly, 90-day retention; includes every LLM call, every OAuth event, every config change.
- **Data retention** — raw MIME off by default; if enabled, S3 lifecycle expires at 395 d; `emails` metadata partitioned at 12 months; summaries kept indefinitely unless user deletes.
- **Local device security (PWA)** — sensitive content rendered via `innerText`, not `innerHTML`; CSP header strict (`default-src 'self'`; no inline scripts); service worker signs registrations; session cookie `HttpOnly Secure SameSite=Lax`.
- **Offline data encryption** — IndexedDB is origin-isolated; we do NOT additionally encrypt because the device OS is the trust boundary and re-encrypting in JS gives pseudo-security. Document this choice.
- **Secure sync** — mutations queue uses idempotency keys (client-generated UUID v7); server rejects duplicates; all API calls use `credentials: 'include'` with CSRF double-submit.
- **Prompt/data redaction** — optional PII redaction before LLM (§6). Off by default for personal use, opt-in via `settings.llm.redact_pii=true`.
- **PII in logs** — structlog processor strips `Authorization`, `Cookie`, `refresh_token`, `access_token`, and email body fields before emission.
- **Threat model (STRIDE, brief):**
  - **Spoofing:** attacker with stolen session cookie — mitigated by short-lived cookies (8 h) + refresh on use + IP/UA anomaly logging.
  - **Tampering:** SQS message tampering — SQS uses IAM, no cross-account access.
  - **Repudiation:** user denies config change — `audit_logs` with actor + timestamp.
  - **Information disclosure:** Anthropic data leak — data processing agreement with Anthropic + opt-in PII redaction + user controls rejection.
  - **DoS:** Gmail quota exhaustion — token bucket + per-account budgets + circuit breaker.
  - **Elevation of privilege:** Lambda execution-role scope — minimal IAM policy per function; `lambda:InvokeFunction` limited to SQS event sources; read-only Lambda runtime FS; reserved concurrency caps runaway invocation.
- **Dependency hygiene** — Dependabot enabled on `main`; `pip-audit` + `npm audit --production` in CI; `trivy fs` on every PR; `trivy image` on every build.

## 12. Release Strategy

- **Semver strictly** — `MAJOR.MINOR.PATCH`. The public contract is: (a) HTTP API under `/api/v1/*`, (b) config schema `user_preferences.config_version`, (c) documented CLI flags, (d) prompt file frontmatter schema.
- **Trunk-based development** — short-lived feature branches off `main`; PR + squash-merge; tags cut from green `main`. Gitflow's `develop`/`release` branches are overkill for a solo/small-team OSS project.
- **Conventional Commits** — `feat / fix / docs / refactor / perf / test / chore / ci / build / revert(!)`. Generates release notes automatically.
- **Tag discipline** — annotated signed tags `vX.Y.Z` only from green `main`; release workflow requires green CI + populated `CHANGELOG.md [Unreleased]` section. `release_metadata` row written on deploy.
- **CHANGELOG** — Keep a Changelog format at `CHANGELOG.md`; CI fails the release if `Unreleased` is empty.
- **Migration policy** — Alembic at `backend/app/db/migrations/`. Forward-only in prod; every migration includes a reversible `downgrade()` for dev. Rule: **additive columns first, data backfill next, drop old columns in the *following* minor** — never in the same release.
- **Config backwards compat** — `user_preferences.config_version` increments on breaking schema change; `PreferencesService` has `upcast_v{n}_to_v{n+1}` functions; rejects un-upcastable versions with `ConfigVersionError`.
- **1.0.0 release criteria** — green CI; eval thresholds met; load-tested at 3 accounts × 300 emails/day for 7 consecutive days on dev; documented rollback; CHANGELOG populated; README + CONTRIBUTING updated.

## 16. Operational Reliability

| Failure mode | Handling |
|---|---|
| Gmail API rate limit (429, quota unit exhaustion) | Token-bucket (250 u/s) + exponential backoff with full jitter (`base=0.5s, cap=30s`); batched `messages.get`. |
| Partial pipeline failure | Idempotent handlers + SQS per-stage; failure in summarize does not lose the classification; `digest_runs.status='partial'` with `degraded_reason`. |
| Retries | tenacity with `stop_after_attempt(3) | stop_after_delay(60)`, `wait_exponential_jitter`. Handlers emit idempotency key so retries are safe. |
| Duplicate ingestion | `UNIQUE(account_id, gmail_message_id)` + content hash for re-threaded messages; `processed_messages` ledger for 7 d. |
| Failed summarization | Message returns to queue; max 5 receives → DLQ; DLQ depth > 0 triggers SNS alarm. |
| Stale sync | `last_incremental_at < now() - 36h` → `sync_cursors.stale=true`; next run does bounded full sync (last 14 d) then resumes. |
| Bad model output | Pydantic `extra="forbid"` + validator exceptions; confidence-gate to `needs_review`; escalate to Opus gated re-eval if enabled. |
| Malformed email | Parser returns `EmailMessage` with `parse_errors[]`; pipeline continues with best-effort metadata; logs per-field errors. |
| Unsubscribe false positive | Never auto-act; confidence floor 0.80; user must click; `dismissed=true` is permanent. |
| Scheduling failure | EventBridge Scheduler target failure alarms via CloudWatch; fan-out Lambda logs surfaced; `digest_runs` missing-row nightly reconciliation job re-enqueues. |
| Account disconnect / re-auth | `oauth_tokens` refresh-failure flagged → `connected_accounts.status='needs_reauth'` → dashboard banner → `/settings/accounts` quick-reconnect. |
| Data poisoning (eval gaming) | Eval set is hand-labeled by the user; changes require explicit commit message tag `eval-set:` and a reviewer on PR. |
| Alarms + DLQ | CloudWatch alarms → SNS → email (free, personal use). DLQ inspected weekly via `/admin/runs?status=failed`. |
| Recovery | One-command restore from S3 pg_dump via `scripts/restore.py`; Alembic forward-only with additive-first rule keeps schema + code compatible across one minor. |

## 18. Risks, Trade-offs, and Recommendations

### Candid challenges to the original ask

1. **"HIPAA-compliant" was wrong and we pushed back.** Personal self-hosted email triage is not a HIPAA workflow. Claiming HIPAA misleads users and over-engineers the system. We are **security-first without being HIPAA-compliant** and say so explicitly in README + `docs/security.md`. Accepting the challenge you invited.
2. **"Processes daily" is under-specified.** Scheduled runs mean user-configurable cron (not hourly, not push); new email arriving at 10 AM waits for the next run unless the user hits manual refresh. We traded push latency for cost + simplicity (no Pub/Sub watch). If real-time is desired in 1.1, add Gmail push via Pub/Sub + SNS bridge.
3. **"Offline support" on iPhone has hard iOS/WebKit limits.** No Background Sync, no guaranteed persistence beyond 7 days of non-use, no Web Push without APNS. We ship **offline reading of cached data + queued mutations** — not background sync. Called out in README.
4. **"Cross-account consolidated digest" + "multi-Gmail support" is simple at small N but contested at high N** (token budgets multiply; quota juggling). Phase-gate to 5 accounts per user in 1.0.0; loosen later.
5. **Supabase vs RDS** — we kept Supabase for 1.0.0 because RDS/Aurora min-ACU pricing (~$43/mo Aurora Serverless v2 at 0.5 ACU) busts the budget for zero gain at personal scale. Alembic-managed portable SQL means the switch is a connection-string change when/if multi-tenant SaaS happens. **This is the single biggest architectural trade-off** — accept cross-cloud latency (~50–100 ms) for cost.
6. **Single-user self-hosted is the 1.0.0 stance.** Open-source means others can fork and run their own instance. A hosted multi-tenant SaaS is a different product with different security, billing, BAA, and ops stories — we do not ship that in 1.0.0.
7. **"No hardcoded values" is discipline, not a library.** It is enforced by code review + config tests that instantiate `PreferencesService` with defaults that match `.env.example`. A lint rule would be false-positive-heavy; we rely on PR review.
8. **"Mobile + offline" is the expensive half of 1.0.0.** It competes with LLM pipeline polish for weeks 10–14. If the schedule slips, cut **visual regression (Chromatic)** and **i18n scaffolding** before cutting offline.

### Key architectural trade-offs (with the accepted side)

| Trade-off | Options | Chose | Why |
|---|---|---|---|
| Compute | **Lambda + SnapStart** (§19.15) | Free tier forever; SnapStart cuts Python cold start to ~300 ms; daily incremental sync fits the 15-min cap. |
| Queue | SQS vs Celery+Redis vs BackgroundTasks | **SQS** | Durable, zero ops, free tier; Redis is $12+/mo always. |
| DB | Supabase vs RDS vs Aurora | **Supabase** | Free tier, RLS free, portable SQL; revisit when multi-tenant or pro-usage. |
| LLM provider | Anthropic direct vs OpenRouter hybrid | **Anthropic direct + OR fallback** | Cost, caching, Batch, typing. |
| Model tiering | One model vs two-tier + gated Opus | **Two-tier + gated Opus** | 13× cost cut on classification; quality retained. |
| Summarization | Sync inline vs Batch API overnight | **Batch overnight (sync fallback)** | 50% off for the bulky path. |
| Frontend styling | Tailwind+shadcn vs Chakra/MUI/Mantine | **Tailwind v4 + shadcn/ui** | Own the sources, lintable, token-true. |
| Forms | rhf+zod vs Formik | **rhf+zod** | Uncontrolled perf + zod shared types. |
| State | Zustand + Query vs Redux + Query vs Jotai | **Zustand + Query** | Smallest useful surface. |
| Offline | Workbox + persistQueryClient vs bespoke | **Workbox + persistQueryClient** | Boring, supported, proven. |
| iOS push | Pursue vs defer | **Defer** | Unreliable pre-16.4 standalone; email digest primary. |
| HIPAA | Comply vs refuse | **Refuse + explain** | Inapplicable; security-first anyway. |

### Honest risks that don't have clean mitigations

- **Anthropic pricing change.** A 2× price hike in 2026 doubles the LLM line. Mitigation: Sonnet → Haiku for summarization with eval regression testing; otherwise accept pass-through to users.
- **Gmail policy change on `gmail.readonly` scope verification.** Verification can take weeks; OSS contributors will hit the unverified-app wall on their personal Cloud projects. Document workarounds in `docs/configuring.md`.
- **iOS Safari PWA policy changes.** Storage quota / eviction rules shift quarterly; the app must tolerate cache-cleared state gracefully — we build for this but it affects UX.
- **Single-user bus factor.** Solo OSS projects stall when the maintainer is out. Minimize via strict docs, high test coverage, and conventional structure so a new contributor can ship within a week.

### 19.9 Security additions

Add to §11 threat model (explicit items I only implied):

- **Prompt injection from email bodies** — email content is untrusted. - **XSS in rendered HTML email snippets** — the PWA never renders email HTML. - `settings.offline.secure_mode=true` enables a user-passcode-derived AES-GCM wrap on `pending_mutations` and persisted summary bodies in IndexedDB. Key in memory only while unlocked; inactivity timeout re-prompts. Documented trade-off: not equivalent to native secure-enclave storage; raises the bar against casual device access, not a determined attacker with device unlock.

Add a **hosted-deployment caveat** (§11 and §17):

### 19.13 Scheduled Maintenance and Retention

The original plan scattered retention across §8 (partitioning), §11 (privacy defaults), and S3 lifecycle rules in §5 without pulling it together into a single executable story. This subsection consolidates database cleanup, partition maintenance, orphan sweeps, and user-initiated deletion into a **single nightly maintenance run**.

#### Orchestration

- **EventBridge Scheduler** rule `briefed-maintenance-nightly` fires at **03:00 UTC** daily (configurable via `user_preferences.maintenance_cron`).
- Target: posts a single message to SQS queue `briefed-maintenance` with `{kind: "nightly", run_id: uuid7()}`.
- Worker handler: `backend/app/workers/handlers/maintenance.py` (new). Runs in a dedicated handler so it cannot compete with the digest pipeline for DB connections; runs inside a single transaction per stage with an advisory lock to prevent overlap.
- Recorded in a new `maintenance_runs` table (schema mirrors `digest_runs`: `id`, `status`, `started_at`, `finished_at`, `stats JSONB`, `error`) for auditability.

#### Stages executed in order

| Stage | What runs | Source of truth |
|---|---|---|
| 1. Partition rotation | Detach + drop `emails` and `audit_logs` partitions older than retention window. Create next month's partition if missing. | `retention_policy_json.emails_metadata_days`, `audit_logs_days` |
| 2. Row-level TTL sweep | `DELETE` rows from `processed_messages` (>7 d), `deferred_llm_calls` (>retry cap AND >30 d), `digest_runs` stuck in `queued`/`running` for >24 h (mark `failed` + surface in `/history`). | `retention_policy_json.idempotency_days`, `deferred_llm_days`, `stuck_run_hours` |
| 3. Summary + classification pruning | Soft-delete (`deleted_at`) then hard-delete `summaries`, `classifications`, `job_matches` orphaned by an `emails` row dropped in stage 1. | FK cascade handles most; this mops up rows whose parent was partition-dropped |
| 4. Unsubscribe dismissal cleanup | Hard-delete `unsubscribe_suggestions` rows where `dismissed=true` AND `dismissed_at < now() - 180d`. | `retention_policy_json.dismissed_suggestions_days` |
| 5. Prompt call log compaction | Aggregate rows in `prompt_call_log` older than 30 d into a daily rollup table (`prompt_call_log_daily` — `date, prompt_version_id, model, calls, tokens_in, tokens_out, cost_usd`), then drop raw rows older than `retention_policy_json.prompt_call_log_days`. | Keeps long-term cost trend visible without keeping every row |
| 6. S3 lifecycle health-check | Sample-probe 10 random keys under `s3://briefed-raw-email/` older than 30 d; assert lifecycle moved them to IA. Alarm if not (indicates broken lifecycle rule). | Defense-in-depth for the AWS-native lifecycle rules |
| 7. Vacuum + analyze | `VACUUM ANALYZE` on tables touched in stages 1–5; `REINDEX CONCURRENTLY` monthly (first run of the month only). | Postgres autovacuum is usually enough but explicit analyze after bulk deletes keeps planner accurate |
| 8. Storage report | Write a `stats` row summarizing bytes reclaimed, rows deleted per table, partitions dropped, alerts raised. Surfaced in `/settings/accounts` as "last maintenance run". | Observability — the user can see cleanup happened |

#### Configurable retention defaults (`user_preferences.retention_policy_json`)

```json
{
  "emails_metadata_days": 365,
  "audit_logs_days": 90,
  "summaries_days": 365,
  "classifications_days": 365,
  "prompt_call_log_days": 180,
  "idempotency_days": 7,
  "deferred_llm_days": 30,
  "dismissed_suggestions_days": 180,
  "stuck_run_hours": 24,
  "raw_mime_enabled": false
}
```

All editable from `/settings/preferences`. The schema is versioned (`retention_policy_version INT`) and breaking changes go through the same `upcast_v{n}_to_v{n+1}` pipeline as the rest of `user_preferences` (§12).

#### User-initiated data deletion

In addition to the nightly job, these explicit user actions bypass retention windows and run synchronously:

- **`DELETE /api/v1/accounts/{id}`** — cascade-deletes all `emails`, `classifications`, `summaries`, `job_matches`, `unsubscribe_suggestions`, `oauth_tokens`, `sync_cursors`, `email_content_blobs`, and associated S3 objects for that account. Revokes the Gmail OAuth token via Google's revocation endpoint. Audited in `audit_logs` with `action='account_deleted'`.
- **`DELETE /api/v1/owner`** — full-tenant wipe: all accounts + all user-owned rows + all S3 artifacts + Cognito/local auth record. Two-step confirmation flow (emails a revocation code, user pastes it). Audited and irreversible.
- **`POST /api/v1/emails/{id}/forget`** — single-email purge (metadata + content blob + S3 object + all derived rows). Useful for one-off "I regret ingesting this" cases.

#### Reliability of the maintenance run itself

- **Idempotent** — every stage keys its work by `maintenance_run.id` + stage name in `processed_messages`; partial completion resumes next night.
- **Advisory lock** — `pg_try_advisory_lock('briefed.maintenance')` prevents overlap with a still-running previous night if a run takes >24 h.
- **Fail-safe** — any stage failing marks the maintenance run `partial`; digest pipeline continues unaffected the next morning; CloudWatch alarm fires if maintenance fails 3 nights in a row.
- **Dry-run mode** — `settings.maintenance.dry_run=true` logs what would be deleted without touching data. Used in staging + for OSS contributors vetting their retention config before enabling.

- **Phase 1 (Foundation)** — add `maintenance_runs` table + stubbed handler + EventBridge rule. No work yet.
- **Phase 2 (Identity + persistence)** — wire `retention_policy_json` into `user_preferences` schema.
- **Phase 8 (Hardening + observability)** — implement all 8 stages, add dashboard widget for last-maintenance status + bytes-reclaimed trend, add dry-run flag, add end-user deletion endpoints with two-step confirmation. Add a chaos test: disable S3 lifecycle, assert stage 6 alarms.

### 19.14 TLS / HTTPS Posture

Consolidates HTTPS enforcement across all layers. Supersedes any plaintext-HTTP language in §5 or §11.

#### Layer-by-layer

| Layer | Transport | Cert source | Enforcement |
|---|---|---|---|
| User browser → CloudFront (PWA) | HTTPS (TLS 1.2+, 1.3 preferred) | ACM | `viewer_protocol_policy = "redirect-to-https"` on every cache behavior; plain HTTP → 301 |
| User browser → Lambda Function URL (API) | HTTPS (TLS 1.2+) | AWS-managed on Function URL | Function URLs are HTTPS-only by construction; no plain HTTP listener exists |
| CloudFront → S3 origin (static assets) | HTTPS | AWS-managed | OAC (Origin Access Control); S3 bucket policy denies non-HTTPS |
| CloudFront → Lambda Function URL origin (API) | HTTPS | AWS-managed | `origin_protocol_policy = "https-only"` + OAC |
| Lambda runtime → Supabase Postgres | TLS (`sslmode=require`) | Supabase-managed | Connection string enforces |
| Lambda runtime → Gemini / Anthropic / Google / AWS APIs | HTTPS | Public CAs | SDK defaults; no override permitted |
| Service Worker registration | HTTPS required | — | iOS/Android refuse SW over HTTP; platform-enforced |

#### Response headers (set via CloudFront response-headers policy + FastAPI middleware)

- `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload` — once the apex domain is on the HSTS preload list, browsers refuse HTTP forever for this host.
- `Content-Security-Policy: default-src 'self'; upgrade-insecure-requests; frame-ancestors 'none'; object-src 'none'; base-uri 'self'` — auto-upgrades any accidental `http://` reference; blocks framing/XSS pivots.
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Permissions-Policy: camera=(), microphone=(), geolocation=(), interest-cohort=()`
- `Cross-Origin-Opener-Policy: same-origin`
- `Cross-Origin-Resource-Policy: same-origin`

#### TLS configuration

- **Minimum version**: TLS 1.2 (`TLSv1.2_2021` security policy on CloudFront). Lambda Function URLs are AWS-managed and negotiate TLS 1.2+ automatically.
- **Preferred**: TLS 1.3 with AEAD ciphers (AES-GCM, ChaCha20-Poly1305).
- **Certificate rotation**: ACM auto-renews 60 days before expiry; CloudFront + ALB references are ARN-pinned in Terraform so rotation is transparent.
- **DNS**: Route 53 A/AAAA alias → CloudFront; `CAA` records pin Amazon as the only issuer authority.
- **HSTS preload submission**: once the domain is stable, submit to `hstspreload.org`. Documented in `docs/operations/tls.md`.

#### Cookies

- Session cookie: `HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=28800`. `Secure` flag means the browser never sends it over HTTP even by accident.
- CSRF double-submit cookie: readable by JS for header echo, but still `Secure; SameSite=Lax`.

#### mTLS — not in 1.0.0

With the canonical Lambda profile (§19.15), there is no VPC-internal plain-HTTP hop — every request is HTTPS end-to-end. mTLS is not applicable for 1.0.0.

#### Terraform wiring

- [infra/terraform/modules/tls.tf](../../Documents/workspace/Misc/Briefed/infra/terraform/modules/tls.tf) — ACM cert resources (apex + wildcard), DNS validation, CAA records.
- [infra/terraform/modules/cloudfront.tf](../../Documents/workspace/Misc/Briefed/infra/terraform/modules/cloudfront.tf) — distribution + `viewer_protocol_policy` + response-headers-policy resource + security headers.
- [infra/terraform/modules/lambda-api.tf](../../Documents/workspace/Misc/Briefed/infra/terraform/modules/lambda-api.tf) — Function URL with `auth_type = "AWS_IAM"` (via CloudFront OAC) + HTTPS-only.
- FastAPI middleware at [backend/app/core/security_headers.py](../../Documents/workspace/Misc/Briefed/backend/app/core/security_headers.py) sets the full header set on every response (CSP, HSTS, nosniff, etc.) so direct Function URL hits during debugging are also hardened.

#### Test cases

- Integration: `curl -I http://app.briefed.example/` → 301 to https.
- Integration: `openssl s_client -connect app.briefed.example:443 -tls1_1` → handshake fails.
- Integration: assert `Strict-Transport-Security`, `Content-Security-Policy`, `X-Content-Type-Options` present on every 2xx response.

### 19.15 Deployment Profile — Lambda + SnapStart + Gemini Flash

**This is the canonical 1.0.0 deployment profile.** **Target steady-state: ~$5–8/month total (AWS + LLM combined).**

#### Compute — Lambda with SnapStart

- **API Lambda** — FastAPI wrapped via `Mangum` ASGI adapter. Single function, container image packaging (Python 3.11 base). Exposed via **Lambda Function URL** with IAM auth off + CloudFront in front (Function URL is free; API Gateway adds unnecessary cost at personal scale). Memory 1024 MB.
- **Worker Lambda** — same container image, different handler entrypoint. Triggered by SQS (event source mapping) from each per-stage queue, batch size 1 with `maximum_batching_window=2s`. Timeout 900 s (15 min max). Memory 1536 MB.
- **SnapStart enabled on both Lambdas** — Python SnapStart (GA late 2025) reduces cold start from ~1–3 s to ~200–300 ms. Free. Initialization code lives at module top-level; post-snapshot hooks (via `@snap_start_hook` decorator or AWS CRaC interface) reset DB connections + boto3 clients on each restore. Zero added cost.
- **No Provisioned Concurrency** — rejected as cost-incompatible (~$11/mo per always-warm instance). SnapStart covers the cold-start concern for personal-scale traffic.
- **Cold-start budget** — API p99 < 500 ms post-snapshot; worker cold start invisible (async SQS path, user doesn't wait on it).

#### Queue + scheduling

- **SQS standard** — per-stage queues `briefed-ingest / classify / summarize / jobs / unsubscribe / digest / maintenance / dlq`. 1M requests/month free forever; personal volume is ~6k/month.
- **EventBridge Scheduler** — same `briefed-daily-<user>` cron rule, targets a tiny fan-out Lambda that enqueues ingestion work. Daily cron at 1/account × 30 days = 30–150 invocations/month. Effectively free.
- **DLQ** — same as §16; alarm on depth > 0 via CloudWatch (alarm count within free tier).

#### Data plane

- **Supabase Postgres free tier** — unchanged. Access via the Supabase **transaction-mode pooler** (`aws-0-<region>.pooler.supabase.com:6543`) — critical for Lambda because it multiplexes short-lived connections. SQLAlchemy `NullPool` on the engine; connection is checked out per request and returned immediately.
- **S3** — same three prefixes (`briefed-raw-email/` optional off, `briefed-digests/`, `briefed-backups/`). 5 GB free year 1 + $0.12/mo after. Lifecycle rules unchanged.
- **Secrets → SSM Parameter Store** — replace AWS Secrets Manager with **SSM Parameter Store Standard tier** (10k parameters free, forever). Parameters: `/briefed/prod/anthropic_api_key`, `/briefed/prod/gemini_api_key`, `/briefed/prod/supabase_url`, `/briefed/prod/supabase_service_key`, `/briefed/prod/google_oauth_client_secret`, `/briefed/prod/session_signing_key`. All stored as `SecureString` using the **AWS-managed `alias/aws/ssm` key** (no customer CMK = $0).

- **Token encryption (OAuth refresh tokens in Postgres)** — unchanged envelope scheme (§11), but the wrapping key is stored as an SSM SecureString (`/briefed/prod/token_wrap_key`) instead of a customer KMS CMK. #### LLM — Gemini Flash primary + Claude Haiku fallback

Replaces §6's "Anthropic direct primary + OpenRouter fallback" for this profile.

- **Primary: `GeminiProvider`** — Google Gemini 1.5 Flash (or newer Flash-tier model available in 2026) via the `google-generativeai` Python SDK. **Paid tier** selected (not free tier) — ~$0.075/1M input, $0.30/1M output. At 18k emails/month:
  - Classification: 18,000 × 400 input + 100 output → 7.2M in + 1.8M out → $0.54 + $0.54 ≈ **$1.08/mo**
  - Summarization: 7,200 × 3,500 in + 300 out → 25.2M in + 2.16M out → $1.89 + $0.65 ≈ **$2.54/mo**
  - **Gemini subtotal: ~$3.60/mo.** Paid tier includes Google's no-training-on-your-data commitment — critical for email content privacy.
- **Fallback: `AnthropicDirectProvider` with Claude Haiku 4.5** — gated invocation only when (a) Gemini returns a schema-validation error twice in a row, (b) Gemini is rate-limited, or (c) classification confidence < 0.55 AND the email matched a `must_read` candidate rule. Hard cap: 100 calls/day project-wide. At ~50 calls/day average = 1,500/mo × 500 tokens avg = ~0.75M tokens → **~$1–2/mo**.

Provider adapter translates the shared `PromptSpec` into whichever mechanism the active provider exposes. Gemini caching is less mature than Anthropic's 1-hour tier — we target a modest ~40% hit rate on classification; cost math above does not rely on caching savings, so any hit is upside.

Already in §7 via `historyId` cursors, but making it explicit for this profile:

#### Honest trade-offs (for the record)

- **Lambda 15-min max** — big Gmail backfills chunked; daily incremental sync fits easily. Documented in the ingestion handler.
- **No persistent DB connection pool** — Supabase pgbouncer transaction-mode pooler handles this. Each Lambda invocation checks out + returns a connection; no keep-alive across invocations.
- **Cold start after deep idle** — even with SnapStart, a request after 2+ h idle may see ~300–500 ms. Acceptable personal-use; not acceptable if you ever want this to be a hosted SaaS.
- **No customer-managed CMK** — if a specific key needed to be revoked to contain a compromise, we can't do it surgically. Mitigation: rotate the SSM-stored wrap key + re-encrypt tokens; documented runbook step.
- **Gemini quality on summarization** — Flash is very good, not quite Sonnet. Acceptable for personal triage; `needs_review` badge covers edge cases.
- **Provider-coupled rate limits** — Gemini paid tier has its own RPM ceiling (higher than free tier but real). Circuit breaker + fallback chain handles spikes.
- **Observability lean** — CloudWatch within free tier means no long log retention; ship a weekly snapshot to S3 if you need >30 day lookback.
- **Atomic rollback** — single Lambda function per role; deploy = replace via Lambda aliases + versions (`aws lambda update-alias --function-name briefed-api --name prod --function-version <previous>`).

### 19.16 Settings + Manual Trigger + Animations

Adds three concrete user-facing capabilities to §10 (UI) and §8 (data model), with motion specs in one place so the animation story stays consistent.

#### 1. Settings page — Gmail account management

Route: `/settings/accounts` (already defined in §10 IA; this subsection specifies behavior).

- **"Add Gmail" primary action** (top-right of the page). Click → `GET /api/v1/auth/google/start?link=true&next=/settings/accounts`. Backend returns a 302 to Google's consent screen; on return, `/oauth/callback` attaches the new refresh token + creates a `connected_accounts` row + navigates back to `/settings/accounts`.
- **Connected account list** — one `<AccountCard>` per row showing:
  - Avatar (Gmail address initial) + `email` + `display_name` + connection date
  - Status chip: `Active` / `Paused` / `Needs reauth` / `Error` (color tokens from `packages/ui/tokens.css`)
  - Sync stats: last successful sync timestamp, emails ingested in last 24 h, daily budget usage %
  - Per-account controls: **`Auto-scan` toggle** (per-account override, falls back to global), **"Scan this account now"** icon-button, **three-dot menu**: Rename, Exclude from global digest, Reconnect, Disconnect
- **Disconnect flow** — confirmation dialog: "This will delete X emails, Y summaries, Z job matches from Briefed. Your Gmail is untouched." → `DELETE /api/v1/accounts/{id}` (§19.13 cascade-delete path).
- **Empty state** — `<EmptyState icon="mail" title="No Gmail accounts yet" cta={<Button variant="primary">Add Gmail</Button>} />` with a short "What happens when you connect?" paragraph explaining `gmail.readonly` scope + recommend-only guarantee (§19.1).

#### 2. Auto-execution toggle (global + per-account)

**Global setting** at `/settings/preferences`:
- `<Field label="Automatic daily scans" description="Briefed will scan your connected Gmail accounts on the schedule below. Turn off to run only manually.">` wrapping a `<Switch>` primitive bound to `user_preferences.auto_execution_enabled`.
- When `false`: EventBridge Scheduler rule for this user is disabled (via `aws scheduler update-schedule --state DISABLED`). The scheduled cron is kept (not deleted) so re-enabling is instant.
- When `false` AND user hasn't triggered a manual scan in >7 days → dashboard banner: `"Auto-scan is off. Last scan: 9 days ago."` with a `[Scan now]` CTA.

**Per-account override** at `/settings/accounts`:
- Switch on the `<AccountCard>` bound to `connected_accounts.auto_scan_enabled` (new column). When explicitly set (not `NULL`), it overrides the global. `NULL` = inherit.
- Fan-out Lambda (§7) checks both: `user.auto_execution_enabled AND (account.auto_scan_enabled IS NULL OR account.auto_scan_enabled = true)`. Only accounts passing both are enqueued.

**Data model addition** (extends §8 + §19.7):
- `user_preferences.auto_execution_enabled BOOLEAN NOT NULL DEFAULT true`
- `connected_accounts.auto_scan_enabled BOOLEAN NULL` (tri-state: `NULL` inherit, `true` force-on, `false` force-off)
- Both auditable: writes produce `audit_logs` rows with `action='toggle_auto_execution'`.

#### 3. Dashboard "Scan Now" button (manual trigger)

Route: `/` (dashboard). Placement: sticky action row at the top, right of the "Today's Digest" heading.

- **Button states** (one `<Button variant="primary" size="lg">` driven by the mutation state):
  - Idle: `[🔄 Scan now]`
  - Hover: subtle lift (translate-y-1 via framer-motion)
  - Pressed: scale(0.97) spring
  - Running: `[⟳ Scanning… 2 of 3 accounts]` with animated spinner and live account counter
  - Success: `[✓ Scanned 47 new emails]` auto-reverts to Idle after 4 s
  - Error: `[⚠ Retry]` with tooltip surfacing `digest_runs.error`
- **API call** — `POST /api/v1/runs` with body `{kind: "manual", account_ids?: string[]}`. Backend:
  1. Checks no active `digest_runs.status IN ('queued','running')` for this user (advisory lock); 409 if one exists.
  2. Creates a `digest_runs` row with `trigger_type='manual'`.
  3. Enqueues fan-out messages to `briefed-ingest` SQS.
  4. Returns `{run_id, accounts_queued}` immediately (202 Accepted).
- **Progress stream** — frontend subscribes via `GET /api/v1/runs/{run_id}/events` (Server-Sent Events over the Lambda Function URL; long-lived responses work under Function URL response-streaming). Events: `ingest_started{account_id}`, `ingest_completed{account_id, count}`, `classification_completed{count}`, `summarization_completed{count}`, `digest_ready`. Backend emits these as SQS messages progress; frontend increments a progress bar.
- **Rate limiting** — hard-limit to 10 manual triggers per user per 24 h (stored on a `manual_run_quota` table; resets rolling). At limit: button disabled with tooltip.

#### 4. Animations — consolidated spec

All animations use `framer-motion` (already in §5). Respects `prefers-reduced-motion` — everything below falls back to instant state transitions when the user has reduced motion enabled.

**Tokens** (already in `packages/ui/tokens.css`):
- `--motion-fast: 120ms`
- `--motion-base: 200ms`
- `--motion-slow: 400ms`
- `--ease-standard: cubic-bezier(.2, .0, .0, 1)`
- `--ease-emphasized: cubic-bezier(.2, .0, 0, 1)`

**Scan Now button:**
- Idle → pressed: `scale: 0.97` over 100 ms.
- Idle → running: spinner fades in (`opacity 0→1` over 120 ms); label crossfade.
- Progress bar under the button: `scaleX` from 0 → 1 keyed to `completed_stages / total_stages`, spring `stiffness=90, damping=20`.
- Running → success: spinner fades out, checkmark draws in via SVG `pathLength: 0→1` over 300 ms with ease-out, then holds 1.6 s, then label crossfades back to idle.

**Auto-scan toggle (`<Switch>`):**
- Thumb translates `x: 0 → 20px` over 150 ms with `spring stiffness=400, damping=30`.
- Track color tween from `--color-border → --color-accent` over 150 ms.
- When toggled off → optional subtle `scale: 1 → 0.96 → 1` bounce for tactile feedback (80 ms).

**Account list on `/settings/accounts`:**
- New account added: `<AccountCard>` enters with `opacity: 0, y: -8 → opacity: 1, y: 0` over 200 ms, ease-emphasized.
- Account removed: exits with `opacity: 1 → 0, height: auto → 0` with `AnimatePresence layout`; staggered siblings collapse up.
- Status chip color change (e.g. `Active → Needs reauth`): background tween over 240 ms + 4 px horizontal shake (3 oscillations) on first paint as attention-grabber.

**Dashboard freshness states** (§19.8 Fresh / Stale / Offline / Sync Failed badge):
- State transition tween: badge pill crossfades + tiny `scale: 0.96 → 1` pulse (120 ms) when the state changes.

**Page transitions:**
- Route-to-route: `opacity: 0 → 1` over 120 ms, `y: 8 → 0` over 180 ms. No slide-right cinema — the goal is subtle reassurance, not motion tax.
- Mobile swipe on `<EmailCard>` (§10): already speced via `drag='x'` with `dragElastic=0.3` and spring-back.

**Dialog / Sheet openers (confirm disconnect, queued-actions sheet):**
- Dialog: backdrop `opacity 0→0.5` over 150 ms; content `opacity 0→1, scale 0.98→1` over 200 ms.
- Sheet (mobile bottom drawer): `y: 100% → 0` spring `stiffness=260, damping=30`.

**Skeleton → content:**
- Skeleton shimmer: linear gradient translates `x: -100% → 100%` over 1400 ms, infinite, delay 100 ms.
- On content load: skeleton fades to 0 opacity over 180 ms while the real content fades in simultaneously (crossfade, no layout shift).

**Implementation note:** wrap all animated primitives in a single `<Motion>` helper at `packages/ui/Motion.tsx` that reads `prefers-reduced-motion` via `useReducedMotion()` from framer-motion and short-circuits transitions. Features pages compose `<Motion>` — no raw `motion.div` outside `packages/ui/`.

#### 6. Mobile (iPhone PWA) behavior

The "mobile app" in 1.0.0 is the installable PWA — no native iOS/Android app (see §2 non-goals). Everything in §19.16 sections 1–5 is required to work on iPhone installed-to-home-screen. Mobile-specific behavior:

**Settings/Accounts on iPhone:**
- `<AccountCard>` renders full-width with stacked metadata (avatar + email on row 1, status chip + sync stats on row 2, actions on row 3). Desktop three-dot menu becomes a **bottom-sheet** (`<Sheet>` primitive from `packages/ui/`) triggered by the row's overflow button — bigger touch targets, safer thumb reach.
- **Swipe-left on `<AccountCard>`** reveals two action buttons: `Pause` (soft — sets `auto_scan_enabled=false`) and `Disconnect` (red — opens confirm dialog). Implemented via framer-motion `drag='x'` with `dragConstraints={{left: -160, right: 0}}`. Same pattern as email-card swipe in §10.
- **"Add Gmail" button** is sticky-bottom on mobile viewports (above the safe-area inset) so it's reachable one-handed. On desktop it stays top-right.

**OAuth on iOS PWA (Add Gmail flow — important gotcha):**
- iOS standalone PWAs cannot reliably complete OAuth in an embedded webview — Google blocks the consent screen with "This browser is not secure" in some configurations.
- Solution: the "Add Gmail" button calls `window.open('/api/v1/auth/google/start', '_blank')` which iOS escalates to external Safari. Google consent completes there, and `/oauth/callback` 302s with a **universal-link-style redirect** back to the PWA scope (`https://app.briefed.example/oauth/callback?...`).
- If the user's iPhone has the PWA installed, iOS opens the callback in the PWA; otherwise it opens in Safari with a banner "Open in Briefed app". Documented in `docs/operations/ios-oauth.md`.
- Post-callback, the PWA polls `GET /api/v1/accounts` on window-focus to pick up the new account — no page reload required.

**Auto-execution toggle on iPhone:**
- `<Switch>` primitive renders at `width: 52px, height: 32px, thumb: 28px` — iOS HIG compliant touch target (§10 already requires ≥44 px tappable zone; the surrounding list row provides it).
- Toggling calls the same `PATCH /api/v1/preferences` or `PATCH /api/v1/accounts/{id}` endpoint. Optimistic update via TanStack Query `onMutate` so the toggle is instant even on slow connections; rollback on 4xx/5xx.
- Full-row tap (not just the thumb) also toggles — larger target for shaky-finger scenarios.

**Scan Now button on iPhone:**
- Moves from sticky-top-right (desktop) to a **pinned card at the top of the dashboard** on mobile, full-width, 56 px tall, prominent primary tint. Label: `[🔄 Scan now]` with the account count below it (`"3 accounts · last scan 2 h ago"`).
- **Thumb-reach placement** — dashboard is the first tab in `<BottomTabBar>` (§10), so the Scan Now card is the first thing the user sees.
- **Running state on mobile** — the card expands in-place (height animates from 56 → 120 px) to show a live per-account progress list. Each row: `[spinner] khirijaganer@gmail.com · 12/47 emails`. Layout uses framer-motion `layout` for smooth resize.
- **Success state** — card snaps back to 56 px, label becomes `[✓ 47 new emails · tap to view]` for 4 s, tapping navigates to `/must-read`.

**SSE + iOS Safari background-tab freeze (real gotcha):**
- iOS Safari (and PWAs) freeze long-lived network connections when the app backgrounds for >30 s. The SSE stream `GET /api/v1/runs/{run_id}/events` will drop.
- Client strategy: `useRunProgress(run_id)` hook prefers SSE; on `error` OR on `visibilitychange → hidden` for >20 s, it falls back to polling `GET /api/v1/runs/{run_id}` every 2 s via TanStack Query `refetchInterval`. On foreground, it re-attempts SSE.
- Run state is always reconstructable from the DB row (`digest_runs.status`, `digest_runs.stats`), so polling is never lossy — just noisier. Documented.
- If the app is fully killed mid-run (user swipes away from the home screen): the run continues server-side (it's async SQS-driven). When the user reopens the PWA, `useRunProgress` subscribes to the in-flight `run_id` via TanStack Query and reconstructs state from the DB.

**Mobile animations — constraints & additions:**
- No hover states on touch devices — CSS `@media (hover: hover)` gates all hover animations so they never fire on iPhone.
- Press states are stronger on mobile: `scale: 0.95` (vs 0.97 on desktop) over 80 ms for a clearer tactile response.
- **Haptics** — iOS does not expose haptic feedback to PWAs as of 2026 (no equivalent of `navigator.vibrate`, which iOS explicitly blocks). Android PWAs get `navigator.vibrate(10)` on successful Scan-completion. Documented trade-off; no-op on iOS.
- Pull-to-refresh on the dashboard triggers the same Scan Now flow (shares the rate-limit counter and the mutation). Progress indicator inherits the 56 px pinned-card animation described above.
- Reduced-motion: iOS users with "Reduce Motion" enabled in Accessibility settings get the `prefers-reduced-motion: reduce` media query automatically. All animations collapse to instant transitions via the `<Motion>` helper (`packages/ui/Motion.tsx`).

**Offline behavior for the new features:**
- **Scan Now when offline** — button is disabled with an inline tooltip: `"Connect to the internet to scan."` We do NOT queue a pending manual-run mutation because the user expects immediate feedback on a trigger action — surprising them with a scan 30 minutes later when they reconnect is worse than refusing now.
- **Auto-execution toggle when offline** — the mutation IS queued in the `pending_mutations` Dexie store (§19.10) and replayed on reconnect. Optimistic UI reflects the intent; the "Sync Failed" badge (§19.8) appears on the toggle if the queued replay fails.
- **Add Gmail when offline** — button disabled; OAuth requires network end-to-end.

**Install prompt behavior:**
- iOS: the existing `<InstallPromptIOS>` primitive (§10) is shown on the dashboard the first time the user visits on iOS Safari non-standalone. Instructions: "Tap Share → Add to Home Screen to install Briefed." Dismissible, stored in Zustand persisted state.
- Android / desktop Chromium: `beforeinstallprompt` event captured; a small `[Install Briefed]` button appears in the dashboard header when eligible.

**Testing additions for mobile:**
- Playwright `mobile-safari` project (iPhone 14 Pro device descriptor) covers: swipe-to-disconnect on `<AccountCard>`, OAuth external-Safari escape + return, Scan Now card expand/collapse animation, SSE-to-polling fallback when backgrounded, pull-to-refresh triggering a scan, reduced-motion bypass of all animations.
- Manual iPhone test pass every release: add-to-home-screen install, standalone-mode viewport, safe-area insets on notched devices, Pro Max vs mini layout check.

### 19.17 Developer commands — `make test`, `make docs`, `make coverage`

Consolidates developer tooling into three top-level Make targets. These are the **canonical commands** run locally and in CI; the existing `/test` and `/make-docs` slash commands wrap them so Claude Code and humans share one surface.

All targets live in the repo-root [Makefile](../../Documents/workspace/Misc/Briefed/Makefile). Each is idempotent, safe to run in parallel, and produces structured output (JSON + human-readable summary).

#### `make test` — run all tests and print a unified pass/fail summary

**What it runs, in order:**
1. Backend lint + type gates (fail fast): `ruff check`, `ruff format --check`, `mypy --strict`.
2. Frontend lint: `npm --prefix frontend run lint`, `npm --prefix frontend run typecheck`.
3. Python unit + integration: `pytest backend/tests -m "not e2e" --junitxml=.artifacts/pytest.xml --json-report --json-report-file=.artifacts/pytest.json`.
4. Frontend unit + component: `npm --prefix frontend run test -- --reporter=json --outputFile=../.artifacts/vitest.json`.
5. Playwright e2e (gated by `PLAYWRIGHT=1` env — skipped by default for fast local loops; CI sets it): `npx playwright test --reporter=json --output=../.artifacts/playwright.json`.
6. Prompt evals (gated by `EVAL=1` env — skipped by default; CI runs only on PRs that touch `packages/prompts/**`): `promptfoo eval -c backend/eval/promptfoo.yaml --output .artifacts/promptfoo.json`.

### 20.3 Security correction — KMS CMK for OAuth wrap key

§19.15 stored the OAuth refresh-token wrap key as an SSM `SecureString` and loaded it into the app's process memory. The peer review correctly flagged this as security theater: an attacker who compromises the app role can read both the ciphertext from Postgres and the wrap key from SSM, so the envelope adds no real boundary on top of platform encryption-at-rest.

**Final design:**
- One **customer-managed KMS CMK** (`alias/briefed-token-wrap`) per environment. ~$1/mo.
- Per-token random 256-bit DEK, AES-GCM ciphertext stored in `oauth_tokens.access_token_ct` / `refresh_token_ct`.
- DEK is wrapped by the CMK via `kms:Encrypt` at write time; unwrapped via `kms:Decrypt` at read time.
- **The wrap-key material never leaves KMS.** The app role has `kms:Decrypt` on this key and nothing else — `kms:GetKeyMaterial`, `kms:ScheduleKeyDeletion`, `kms:DisableKey` are reserved for the deploy/admin role.
- Every unwrap is logged in CloudTrail. Anomalous decrypt rates fire a CloudWatch alarm.
- **Containment**: revoking the app role's `kms:Decrypt` permission instantly bricks all token reads without requiring a re-encryption sweep. That is the trust boundary the SSM design lacked.
- DEK rotation on re-auth (unchanged); CMK rotation enabled with `EnableKeyRotation = true` (annual, AWS-managed, transparent).
- Cost: ~$1/mo + `~$0.03/10k decrypt calls` — at ~150 unwraps/day this is rounding error.

Updated cost table line: replace "KMS AWS-managed keys $0" with **"KMS customer-managed CMK ~$1/mo"**. New steady-state: **~$7–10/mo total** (was $6–9). Worth $1/mo for a real security boundary.

### 20.10 Content-at-rest encryption — shrink Supabase's view to metadata only

§20.3 closed the OAuth-token trust boundary against a Supabase compromise. This subsection closes the *content* boundary: LLM-generated summaries and rationale text get the same envelope-encryption treatment so a Supabase-side read yields metadata only (subjects, senders, timestamps, labels) — never email bodies or summarized content.

**Threat addressed.** A malicious or compelled Supabase operator with table-level access reads plaintext rows. Today, those rows include `summaries.body_md` (a faithful summary of email content) and `classifications.reasons` (LLM rationale text that paraphrases content). Encrypting them at the application layer means the row is opaque without a `kms:Decrypt` call against a CMK in the operator's *own* AWS account.

**Encrypted columns.** Per-row envelope-encrypted; same AES-GCM + wrapped-DEK pattern as OAuth tokens (§20.3):

| Table | Plaintext column → ciphertext columns |
|---|---|
| `summaries` | `body_md` → `body_md_ct BYTEA NOT NULL`, `body_dek_wrapped BYTEA NOT NULL` |
| `email_content_blobs` | `plain_text_excerpt` → `plain_text_excerpt_ct BYTEA`, `plain_text_dek_wrapped BYTEA` |
| `classifications` | `reasons JSONB` → `reasons_ct BYTEA`, `reasons_dek_wrapped BYTEA` |
| `job_matches` | `match_reason TEXT` → `match_reason_ct BYTEA`, `match_reason_dek_wrapped BYTEA` |

**Not encrypted** (deliberate — these are metadata Supabase can already infer from row counts/sizes, and encrypting them breaks core indexing): `emails.from_addr`, `to_addrs`, `subject`, `snippet`, `labels`, `internal_date`; `classifications.label` + `score`; `job_matches.title`, `company`, `location`, `salary_*`. The `subject` and `snippet` fields are a deliberate trade — encrypting them would block the dashboard list views; they are the smallest leak in the encrypted set and you can opt them in via `settings.security.encrypt_subject=true` if your threat model demands it.

**Second KMS CMK.** A separate CMK `alias/briefed-content-encrypt` (in addition to `alias/briefed-token-wrap` from §20.3). Reasons for two keys, not one with encryption-context separation:
- **Independent revocation.** Revoking content reads (e.g., post-incident scoping) must not break OAuth refresh, and vice versa.
- **Independent rotation cadence.** Token-wrap key rotates on a security cadence; content key rotates on a data-lifecycle cadence.
- **Independent audit.** CloudTrail filters by key ARN — content-decrypt anomalies are distinguishable from token-decrypt anomalies in alarms.
- Cost: +~$1/mo. Justified for a real boundary, same logic as §20.3.

**Encryption context** (KMS feature — bound into every `kms:Encrypt` / `kms:Decrypt` call): `{user_id, table, row_id, purpose: "content"}`. KMS rejects a Decrypt request whose context doesn't match the Encrypt context, so an attacker cannot swap a ciphertext from one row into another row's slot.

**Crypto helper.** New module [backend/app/core/content_crypto.py](../../Documents/workspace/Misc/Briefed/backend/app/core/content_crypto.py):

```python
async def encrypt_content(plaintext: bytes, ctx: EncryptionContext) -> EncryptedBlob: ...
async def decrypt_content(blob: EncryptedBlob, ctx: EncryptionContext) -> bytes: ...
```

Generates a 256-bit DEK per row, AES-GCM-encrypts the plaintext, calls `kms:Encrypt` to wrap the DEK with `alias/briefed-content-encrypt` and the supplied context, returns `(ciphertext, wrapped_dek, nonce, auth_tag)` packed into one `BYTEA` column with a 1-byte version prefix for forward compatibility.

**Repository layer is the boundary.** `SummariesRepo`, `ClassificationsRepo`, `JobMatchesRepo`, `EmailContentBlobsRepo` transparently encrypt on write and decrypt on read. Pydantic boundary models (`Summary`, `Classification`, etc.) hold plaintext; SQLAlchemy models hold ciphertext. **No service-layer code changes** — the encryption is invisible above the repo layer.

**Performance.** Each encrypted-field read = one `kms:Decrypt` (~5–10 ms p50 in-region) + one AES-GCM decrypt (~µs). Daily digest opens ~50 summaries → ~250–500 ms total KMS latency, parallelizable via `asyncio.gather`. Mitigation: a per-Lambda-warm-window LRU cache (max 200 entries, keyed by `sha256(wrapped_dek)`) stores unwrapped DEKs in memory; cuts repeat-read latency to AES speed. Cache is dropped on cold start (no cross-invocation persistence) so a SnapStart restore re-pays the KMS round-trip on first read — acceptable.

**What this breaks (honest list).**
- **Server-side full-text search on summaries** is gone. The plan didn't require it for 1.0.0; if needed in 1.1, options are (a) deterministic searchable encryption (HMAC tokens over keywords), (b) client-side filtering on the already-decrypted offline cache, or (c) move search to a separate plaintext index in S3/OpenSearch with explicit user opt-in.
- **`pg_dump` backups contain ciphertext.** Restore drill must verify KMS access *before* restoring data; runbook update at `docs/operations/restore.md`. Test as a Phase 8 chaos drill: restore the dev DB to a fresh project, confirm content is unreadable until the CMK grant is in place.
- **Frontend offline cache (Dexie / IndexedDB) still holds plaintext** — encryption stops at the device boundary per §11. The optional `settings.offline.secure_mode` (§19.9) is the lever for users who want a passcode-derived wrap on the IDB store too.
- **Adding a new encrypted column later is a migration + backfill** — not a runtime change. Build the helper to make this mechanical; document the playbook in `docs/operations/add-encrypted-column.md`.

**What this still does NOT protect against.**
- Anthropic / Gemini reading content during inference — they receive plaintext by definition; that's a different boundary handled by the no-training-tier choice (§19.15) and optional PII redaction (§6).
- A compromise of the user's AWS account itself — if an attacker gets `kms:Decrypt` on the content CMK, they can decrypt at will. Mitigation: scope the grant to the worker Lambda's execution role only; never to a developer IAM user.
- Traffic analysis — Supabase still sees query patterns and row counts. Not addressable without a different data plane.

**Phase plan impact.**
- **Phase 0** — add `alias/briefed-content-encrypt` CMK to [infra/terraform/modules/kms.tf](../../Documents/workspace/Misc/Briefed/infra/terraform/modules/kms.tf); add `kms:Encrypt` + `kms:Decrypt` permission scoped to this key on the worker + api Lambda execution roles.
- **Phase 1** — add `content_crypto.py` helper alongside `core/security.py` (envelope crypto for tokens). Same primitive, different CMK alias and encryption context.
- **Phase 2** — `classifications.reasons_ct` lands here; `ClassificationsRepo` encrypts on write from day one.
- **Phase 3** — `summaries.body_md_ct` and `email_content_blobs.plain_text_excerpt_ct` land here; `SummariesRepo` encrypts on write. **Build encrypted from day one — do not ship a plaintext-summaries milestone and migrate later.** Backfilling encryption on a populated DB is painful; greenfield avoids it.
- **Phase 4** — `job_matches.match_reason_ct` lands here.
- **Phase 8** — KMS revocation drill expands: revoke `kms:Decrypt` on the content CMK → assert summaries become unreadable in the UI → restore → assert recovery. Backup-restore drill verifies CMK grant is required *before* the `pg_restore` returns useful rows. Add a CloudWatch alarm on `kms:Decrypt` rate against the content CMK exceeding 10× the rolling 7-day baseline (anomalous bulk read).

**Trade-off summary.** +$1/mo, +1 helper module, +~5 ms per summary read, loss of server-side summary search (not in 1.0.0 scope anyway). In exchange: a Supabase-side data exfiltration yields metadata, not content. For a personal email triage agent storing months of summarized correspondence, that boundary is worth the cost.
